# 藏宝阁多账号价格监控（骨架）

100+ 物品 × 各服务器，每天 2 次最低价，10 账号分摊负载。核心爬取逻辑已对真实接口实测跑通。

## 能不能做到？——能。

| 方案 | 单物品/轮请求数 | 100物品×2/天 | 拿得到<300价 |
|---|---|---|---|
| 跨服(贵价物品) | ~100页 | ~2万/天 | ❌ 仅≥300 |
| 本服(便宜物品) | 333服=333 | ~6.7万/天 | ✅ 真实最低 |
| **混合(默认)** | 自动分流 | **~2–4万/天** | ✅ |

- **账号承载力**：实测单线程~1次/秒连跑 333 次 0 验证码；4路并发~150次就触发验证码。所以**每账号必须单线程慢速**，安全日量约 5k–10k 次。
- **10 账号 = 5–10 万/天容量**，混合方案很从容，纯本服也放得下。

## 架构

```
scheduler(每天2次) → 任务队列 → 账号池(N个worker,各自单线程慢速)
  贵价物品 = 1个"跨服扫描"任务
  便宜物品 = 按服务器分片成多个"本服"任务，分摊到不同账号并行
  遇验证码 → 该账号冷却+告警，任务退回由别的账号接管
  → SQLite price_history(物品,服务器,价格,时间) → 看板/导出
```

## 快速开始

```bash
pip install requests
cp config.example.json config.json     # 填入账号cookie + 物品清单
python3 cbg_monitor.py servers          # 拉取服务器列表(免登录,缓存到 servers.json)
python3 cbg_monitor.py run              # 立即跑一轮
python3 cbg_monitor.py serve            # 常驻，按 run_at 每天定时跑
```

也可不用 `serve`，直接交给系统 cron：
```
0 9,21 * * *  cd /path/to/cbg_monitor && python3 cbg_monitor.py run >> run.log 2>&1
```

## 关键参数（config.json → settings）

- `per_request_delay` / `jitter`：每次请求间隔（秒）。**越大越稳**，是防验证码的核心旋钮。
- `per_server_chunk`：本服任务的分片大小，决定一个物品摊给多少账号并行。
- `overall_page_cap`：跨服最多翻几页（接口硬上限 100 页 / 最便宜 1500 条）。
- `captcha_cooldown_sec`：账号触发验证码后的冷却时长。

## 物品清单怎么填（type_ids 怎么拿）

1. 浏览器登录藏宝阁，进"召唤兽搜索"，类型框输入物品名搜一次。
2. F12 → Network → 找 `recommend.py` 请求 → 看 `type` 参数即类型ID。
   - 例：持国巡守 = `102242,102245`；只要正常版填 `102242`，变异版是 `102245`。
3. 便宜物品(常<300)设 `mode: per_server`，贵价物品设 `mode: overall`。

## 必读限制

- **验证码我不能自动绕过**（属人机检测绕过，不做）。引擎遇验证码会暂停该账号并告警，**需人工到浏览器解一次**；靠慢速 + 多账号轮换把频率压到很低。
- **cookie 会过期**：失效账号会被标记 `dead`，需重新登录并更新 `config.json` 的 cookie。
- **目前只验证了 pet(召唤兽)品类**。装备/锦衣/灵饰等的 `search_type` 和类型体系不同，需按同样方法（在对应搜索页抓一次接口）扩展 `Account` 里的查询函数。
- 自动化高频抓取需自担合规风险，请控制频率、尊重对方服务器。
